# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

DramaCommerce AI is a React Router 8 full-stack app. A merchant uploads a product image plus a short brief (audience, mood, platform, duration); the app runs a Qwen-powered AI "showrunner" pipeline — Analyze (vision) → Story → Director → Prompt → Critic → Editor — that produces a story concept, hook, voice-over, a 5-scene storyboard with video prompts, an editing timeline, a social caption/CTA, can kick off real video renders for each scene via Alibaba Cloud's Wan model (text-to-video by default, or image-to-video using the real product photo for scenes the Director Agent marks as reference-eligible; each scene's AI-written voice-over line is synthesized via DashScope TTS and muxed onto the clip), and stitches the 5 successful clips into one final drama ad with ffmpeg.

## Commands

Package manager is **pnpm** (see `pnpm-lock.yaml` and the Dockerfile).

- `pnpm dev` — start the dev server (React Router dev, HMR) at `http://localhost:5173`
- `pnpm run typecheck` — runs `react-router typegen` then `tsc` (regenerate route types before type-checking after changing `app/routes.ts` or route loader/action signatures)
- `pnpm run build` — production build
- `pnpm run start` — serve the production build (`react-router-serve ./build/server/index.js`)

There is no lint script configured in `package.json`. `pnpm run test` runs a small unit test suite (Node's built-in test runner via `tsx`) covering `app/services/storage/*` (the media storage abstraction's local and OSS drivers, OSS tested against an injected fake client, not a real bucket) and `app/services/outbox.server.test.ts`/`app/services/project-store.server.test.ts` (transactional outbox + idempotency behavior, against a real local Postgres — no test suite in this repo mocks Postgres itself, see [Reliability & job coordination](#reliability--job-coordination)). No other code paths have automated tests.

`scripts/video-worker.mjs` (`pnpm run worker:video`) shells out to `ffmpeg` for the final-video stitch step — it must be installed and on `PATH` wherever the worker runs (already handled in the Docker image via `apk add ffmpeg`, but needs a manual install for local dev).

`scripts/showrunner-worker.mts` (`pnpm run worker:showrunner`) runs the Story/Director/Prompt/Editor agent pipeline as a background job. Unlike `video-worker.mjs` (plain `.mjs`, duplicates Wan/TTS API calls in raw JS), this worker imports the app's real TS modules directly (`services/showrunner.server.ts`, `services/project-store.server.ts`) via the `~/*` path alias — no duplicated agent logic. In development, `pnpm run worker:showrunner` runs the `.mts` source directly via `tsx`. In production, the Docker build instead runs `pnpm run build:worker` (esbuild, bundling the worker and its `~/*`-aliased imports — resolved from `tsconfig.json`'s `paths` — into one file, `build/worker/showrunner-worker.mjs`, with `node_modules` packages left external), and the container runs `pnpm run start:worker`, a plain `node` invocation with no `tsx`/TypeScript/`app/` source needed at runtime. `tsx` and `esbuild` are both devDependencies for this reason — only `pnpm run build`'s output ships to production, never the compiler/loader itself.

## Environment

Copy `.env.example` to `.env`. Three separate Alibaba Cloud DashScope surfaces are used, each with its own base URL var (even though TTS and Wan usually share the same workspace URL in practice):

- `DASHSCOPE_API_KEY`, `QWEN_BASE_URL` (includes `/compatible-mode/v1` — OpenAI-compatible chat completions), `QWEN_MODEL` — for the showrunner planning step. `QWEN_VISION_MODEL` (default `qwen3-vl-flash`, same base URL/key) is used only by the Analyze Agent's vision call.
- `DASHSCOPE_VIDEO_BASE_URL` (no `/compatible-mode/v1` suffix — uses `/api/v1/services/...` and `/api/v1/tasks/...`), `WAN_VIDEO_MODEL`, `WAN_VIDEO_RESOLUTION`, `WAN_VIDEO_RATIO`, `WAN_VIDEO_DURATION` — for Wan text-to-video generation. `WAN_VIDEO_I2V_MODEL` (default `wan2.1-i2v-turbo`) is used instead, on the **same endpoint**, for scenes where `useProductReference` is true — the request swaps `prompt`-only input for `{prompt, img_url}` and drops the `ratio` param (i2v derives aspect ratio from the reference image).
- `DASHSCOPE_TTS_BASE_URL` (same `/api/v1/services/...` style), `DASHSCOPE_TTS_MODEL` (default `qwen3-tts-flash`), `DASHSCOPE_TTS_VOICE` (default `Cherry`) — for scene voice-over synthesis. Uses the same `DASHSCOPE_API_KEY`.

If Qwen env vars are missing or the Qwen call fails, generation returns an error and no project is created. Do not reintroduce automatic mock fallback for production generation.

`MEDIA_STORAGE_DRIVER` (`local` default or `oss`) selects the media storage driver — see the Media storage section below. `oss` requires `OSS_REGION`, `OSS_BUCKET`, `OSS_ACCESS_KEY_ID`, `OSS_ACCESS_KEY_SECRET`; `OSS_ENDPOINT`, `OSS_PUBLIC_BASE_URL`, and `OSS_SIGNED_URL_EXPIRES_SECONDS` are optional.

## Architecture

### Pipeline: brief → showrunner (async) → project → video

The full pipeline is visualized as a live "Agent Timeline" spanning two pages: `Analyze → Story → Director → Prompt → Critic → Editor` on `/projects/new/:jobId`, then `Render → Stitch` on `/projects/:id`. Both halves follow the same shape — write a Postgres row + outbox event, `worker:outbox` publishes to BullMQ, a worker updates a Postgres status column stage by stage, the page polls and re-renders via the shared `app/components/agent-timeline.tsx` — because that's the only way to show live per-stage progress without inventing streaming infra this codebase doesn't otherwise use. `app/components/agent-timeline.tsx` also supports an `orientation` prop (`"vertical"` default, `"horizontal"`) — `/projects/:id` renders it horizontally in a full-width section above the two-column layout; `/projects/new/:jobId` keeps the original vertical layout.

1. `routes/projects.new.tsx` (`action`) collects the form/multipart submission, saves the uploaded image via `services/image-upload.server.ts`, then calls `services/project-store.server.ts#createShowrunnerJobWithOutbox` — inserts the `showrunner_jobs` row and a matching `outbox_events` row (`job_name: "showrunner.generate"`) in one Postgres transaction, and redirects to `/projects/new/:jobId`. It does **not** call `generateShowPlan` itself, and does **not** call BullMQ directly — see [Reliability & job coordination](#reliability--job-coordination).
2. `scripts/showrunner-worker.mts` (`pnpm run worker:showrunner`) picks up the job and calls `services/showrunner.server.ts#generateShowPlan(brief, onStageChange)`, which orchestrates six Qwen-powered stages in order, invoking `onStageChange` before each one so the worker can write the current stage (`ANALYZING` / `STORY` / `DIRECTING` / `PROMPTING` / `CRITIQUING` / `EDITING`) to `showrunner_jobs.status`. Each agent validates its JSON through `services/showrunner-validator.server.ts` before the next stage runs:
   - `agents/analyze-agent.server.ts` — a vision call (`services/qwen.server.ts#callQwenVisionJson`, `QWEN_VISION_MODEL`) on the uploaded photo, base64-inlined via the media storage abstraction (`services/image-upload.server.ts#readUploadedImageAsDataUrl`, which delegates to `services/storage/media-storage.server.ts` — required even in production, DashScope fetches by URL or accepts inline data, and inlining avoids ever needing the upload to be publicly reachable regardless of whether storage is local disk or OSS). Returns category/colors/material/branding/quality/`canUseAsReference`/issues; feeds into Story and Director as grounding context.
   - `agents/story-agent.server.ts` — brief + analysis → `StoryPackage` (concept, hook, voice-over). Right after this stage, `services/story-bible.server.ts#buildStoryBible(brief, analysis, story)` condenses brief+analysis+story into a compact `StoryBible` (`productFacts`/`visualStyle`/`storyCore`/`constraints`) that every downstream agent below reads instead of the raw objects — those don't need fields like `imageUrl`/`showProductOverlay`, and re-serializing all three full objects into five separate prompts was pure repeated context. Raw `brief`/`analysis` are still passed into the agent functions where needed for local skill computation (`services/skills/*`), which costs no tokens since it's not sent to Qwen.
   - `agents/director-agent.server.ts` — brief + analysis + Story Bible → `DirectedScene[]` (5 fixed scenes, each with `camera`/`emotion`/`useProductReference`). `useProductReference` may only be true when `analysis.canUseAsReference` (`storyBible.visualStyle.canUseAsReference` in the prompt) is true **and** the scene's composition could plausibly continue from the actual photo (normally just the final reveal/hero scene) — Wan i2v uses the photo as a literal first frame, so forcing it onto an unrelated b-roll shot produces broken video.
   - `agents/prompt-agent.server.ts` — scenes + Story Bible → `StoryboardScene[]` (adds a Wan-ready `videoPrompt` per scene). Uses Qwen function-calling (`services/qwen.server.ts#callQwenJson`'s `tools`/`toolHandlers` params) to call a local `get_wan_video_constraints` tool for the real configured resolution/ratio/duration before writing prompts, instead of guessing. Note: Qwen's OpenAI-compatible endpoint hangs indefinitely if `response_format: json_object` is combined with `tools` in the same request (confirmed live) — `callQwenJson` only sets `response_format` on tool-free calls. Accepts an optional `revisionNotes` param for the Critic's revision pass below.
   - `agents/critic-agent.server.ts` — storyboard + Story Bible (+ raw `analysis` for its own skill checks) reviews the storyboard for pacing/coherence and sanity-checks `useProductReference` choices against each scene's actual `videoPrompt`. Returns `{approved, notes?}`; if not approved, `generateShowPlan` re-runs the Prompt Agent once with the notes — bounded to exactly one revision pass, no second critique call.
   - `agents/editor-agent.server.ts` — storyboard + Story Bible → `EditorPackage` (timeline, caption, CTA). Reads `storyCore.hook` from the Story Bible and is instructed to open/riff on it in the caption, so the caption doesn't drift to a different angle than the Hook the merchant sees and can edit.
3. On success the worker calls `saveProject` and marks the job `SUCCEEDED` with `projectId` set; `/projects/new/:jobId`'s loader redirects to `/projects/:id` once it sees that. On failure (BullMQ retries exhausted) the worker marks the job `FAILED` with a friendly error via `services/qwen.server.ts#getQwenErrorMessage` and deletes the uploaded image (mirrors the old synchronous failure cleanup) — no project is created either way.
4. `routes/projects.tsx` lists saved projects; `routes/projects.$projectId.tsx` shows one project's full plan and drives Wan video generation for any/all scenes. `intent=create-video-task` / `intent=create-all-video-tasks` call `services/project-store.server.ts#createVideoGenerationWithOutbox` (not `services/wan-video.server.ts` — see its dead-code note below), which atomically upserts the `video_jobs` row (minting a fresh `generation_id`) and its outbox event, but only if the scene isn't already actively generating (`ON CONFLICT ... DO UPDATE ... WHERE status NOT IN (active statuses)` — a duplicate click is a no-op). `intent=refresh-video-task` still calls `services/wan-video.server.ts#queryWanVideoTask` directly (a read, not a job creation, so it's outside the outbox pattern). `scripts/video-worker.mjs` reads each job's `useProductReference` flag from its payload: if true, it reads the product image through `scripts/lib/media-storage.mjs` (a plain-JS mirror of `services/storage/*`, since this worker runs via plain `node`, not `tsx`/esbuild — see the note on this file's duplication pattern below), base64-encodes it, and calls Wan with `WAN_VIDEO_I2V_MODEL` + `img_url` instead of `WAN_VIDEO_MODEL` + `prompt`-only — but only after confirming the row's `generation_id` still matches the payload's (see [Reliability & job coordination](#reliability--job-coordination)). When it polls a scene to `SUCCEEDED`, it synthesizes that scene's `voiceOver` line via DashScope TTS and muxes the audio onto the clip with `ffmpeg` before saving — the raw Wan clip has no audio track at all. If TTS or muxing fails, the worker falls back to the silent Wan clip rather than failing the job (the Wan generation cost is already spent). The product-image overlay (small watermark-style thumbnail in the corner) is only applied when `useProductReference` is false — a reference-image scene already has the real photo baked into frame one, so stamping it on top again would be redundant — and only when the merchant hasn't opted out via the `showProductOverlay` checkbox on `/projects/new` (`ProductBrief.showProductOverlay`, threaded through `VideoCreateJobData.showOverlay`; `undefined`/missing is treated as `true` for backward compatibility with projects saved before this field existed). Once all 5 scenes reach `SUCCEEDED`, `intent=create-stitch-task` calls `services/project-store.server.ts#createStitchGenerationWithOutbox` (same idempotent-upsert shape, minting `stitch_generation_id`); the worker downloads the 5 (now-narrated) clips, concatenates them with `ffmpeg` (stream-copy first, re-encode fallback if codecs mismatch), and saves the result through the storage abstraction (`final-videos/<projectId>/<uuid>.mp4`). Stitch status/output is tracked in the `final_videos` table (one row per project) via `services/project-store.server.ts#saveFinalVideo` (called by the worker directly, not through the outbox — only *creating* a stitch generation goes through it) — `video_url` there (and on `video_jobs`) stores the storage key/reference, not a resolved URL; see the Storage section below.

Every `callQwenJson`/`callQwenVisionJson` call (`services/qwen.server.ts`) accepts an optional `onUsage` callback reporting `{promptTokens, completionTokens, totalTokens}`, summed across BullMQ tool-calling rounds. `generateShowPlan` tags each call with its stage/model and collects them into `ShowPlan.tokenUsage` (optional field, no migration needed — `show_plan` is jsonb), rendered as a table on `/projects/:id`.

`wan-video.server.ts#createWanTextToVideoTask` (the TS version) is currently unused dead code — the actual Wan task creation logic (including the i2v branch above) lives only in `scripts/video-worker.mjs`, which duplicates it in plain JS rather than importing app services (a deliberate, pre-existing pattern for that worker, not something this pipeline change altered).

Editor Agent's `timeline`/`caption`/`cta` output is still display-only — the actual `ffmpeg` stitch step is a plain concat and does not burn in subtitles or append an end-of-video packshot from `EditorPackage`. That integration (making the render actually consume Editor Agent's output) is a deliberately separate, not-yet-started follow-up.

### Reliability & job coordination

HTTP routes never call BullMQ directly — `app/services/outbox.server.ts` defines a Postgres-backed transactional outbox (`outbox_events` table: `queue`/`job_name`/`job_key` (unique)/`payload`/`status`/`attempts`/`available_at`/`processed_at`/`last_error`), and `services/project-store.server.ts`'s `createShowrunnerJobWithOutbox`/`createVideoGenerationWithOutbox`/`createStitchGenerationWithOutbox` are the only places that write to it — each opens a `db.transaction()`, upserts the domain row, and inserts the outbox event, so both commit atomically or neither does. `scripts/outbox-dispatcher.mts` (`pnpm run worker:outbox` / `start:outbox` in prod, bundled the same way as `showrunner-worker.mts`) is the only process that calls `queue.add()`: it polls for `PENDING` events, claims each with `SELECT ... FOR UPDATE SKIP LOCKED` in its own short transaction (safe for multiple dispatcher instances — a row already locked by another instance is skipped, not blocked on), and marks it `DELIVERED` only after `queue.add()` resolves, in the same transaction. `job_key` doubles as BullMQ's deterministic `jobId` — note the delimiter is `_`, not `:` (`showrunner-generate_<id>`, `video-create_<projectId>_<scene>_<generationId>`, `video-stitch_<projectId>_<stitchGenerationId>`), because BullMQ rejects custom job IDs containing `:` (its own internal Redis key delimiter) — discovered via the dispatcher's own smoke test, not from the original design.

Idempotency for "start work" actions is enforced by a fresh UUID (`generation_id` on `video_jobs`, `stitch_generation_id` on `final_videos`) minted only when a *new* generation actually begins, via a single atomic `INSERT ... ON CONFLICT (...) DO UPDATE ... WHERE status NOT IN ('QUEUED','PENDING','RUNNING','UNKNOWN')` — Postgres serializes concurrent writers on that row itself, so no separate row lock is needed for this check; if the row is already active the upsert is a no-op and the existing row is returned (`created: false`), which callers use to skip `recordUsageEvent` so a duplicate click never double-bills quota. Showrunner generation instead reuses a server-generated idempotency token (minted in `/projects/new`'s loader, round-tripped through a hidden field) as `showrunner_jobs.id` itself, so a duplicate POST hits a primary-key conflict.

Every provider-call/state-write in `scripts/video-worker.mjs` is scoped to the `generation_id`/`stitch_generation_id` carried in its BullMQ job payload, via conditional `UPDATE ... WHERE ... AND generation_id = $x` checked against the query result's row count — 0 rows means a newer generation has since superseded this job, and the worker logs (prefixed `[stale]`, with project/scene/generation identifiers) and exits successfully without touching Wan or the row. `video.poll` jobs are *not* routed through the outbox (deliberate scope decision, not an oversight) — they're a worker's own self-rescheduled continuation (`queue.add("video.poll", ..., {delay})`, same as before this change) rather than an HTTP-triggered "start work" event, and the conditional-`UPDATE` guard above already fully covers "an old poll job must not overwrite a newer generation" regardless of how the poll job was enqueued; routing every 30-second poll tick through outbox-insert → dispatcher-pickup → publish would add latency for no correctness benefit. Poll jobs still get a deterministic `jobId` (`video-poll_<projectId>_<scene>_<taskId>_<attempts>`) as cheap insurance against BullMQ-level redelivery.

The one gap that can't be closed without provider-side idempotency keys (Wan's API has none): a worker crashing after calling Wan but before saving the returned `task_id` will, on BullMQ's at-least-once redelivery, call Wan again with no way to know a task already exists. Mitigated via a conditional `QUEUED → RUNNING` transition scoped to `generation_id` immediately before the Wan call (`claimGenerationForWanCall` in `video-worker.mjs`): a fast retry (the crashed attempt's job redelivered) finds the row already `RUNNING` and skips re-calling Wan; `WAN_TASK_STALE_RUNNING_GRACE_MS` (default 5 min) bounds how long a row can sit `RUNNING` before a retry is allowed to proceed anyway, trading a small residual duplicate-call risk for not leaving genuinely stuck jobs stuck forever.

The dispatcher also runs periodic retention cleanup (`OUTBOX_CLEANUP_INTERVAL_MS`, default hourly) deleting `DELIVERED` events older than `OUTBOX_DELIVERED_RETENTION_MS` (default 24h) and `FAILED` events older than `OUTBOX_FAILED_RETENTION_MS` (default 14d) via `services/outbox.server.ts#cleanupOutboxEvents`. `/health`'s `outbox` check (`services/outbox.server.ts#getOutboxStats`) reports `pendingCount`/`oldestPendingAgeSeconds`/`failedCount` and flags unhealthy once the oldest pending event exceeds `OUTBOX_HEALTH_STALE_THRESHOLD_SECONDS` (default 120s) — the only available proxy for "is the dispatcher actually running", since there's no direct way to ping a separate process.

### Storage

`services/project-store.server.ts` persists projects, showrunner jobs, and video jobs to Postgres through Drizzle (`app/db/schema.ts`, `services/db.server.ts`). Migrations live in `drizzle/` and should be applied with `pnpm run db:migrate` before starting web or worker processes. `projects.show_plan` is stored as JSONB, `showrunner_jobs` stores the pending brief (`brief_json`), current pipeline status, error, and the resulting `project_id` once saved, `video_jobs` stores per-scene provider, queue job ID, Wan task ID, status, attempts, polling timestamps, output URL, and error metadata, and `final_videos` stores the stitched-output status/URL per project.

Media (`brief.imageUrl` inside `show_plan`/`brief_json`, `video_jobs.video_url`, `final_videos.video_url`) is stored as a **storage reference** (a bare object key like `product-images/<uuid>.jpg`, or a legacy `/uploads/<uuid>.jpg` path from before this abstraction existed) — never a resolved or signed URL, so nothing short-lived ends up in Postgres. `getProject`/`listProjects`/`deleteProject` in `project-store.server.ts` return these raw references as-is (needed for deletion and for the Analyze Agent's storage read); `getProjectForDisplay`/`listProjectsForDisplay` additionally resolve every reference to a browser-usable URL via `services/storage/media-storage.server.ts#getMediaStorage().resolveUrl()` and are what the three routes that render `<img>`/`<video>` (`routes/dashboard.tsx`, `routes/projects.tsx`, `routes/projects.$projectId.tsx`'s loader) actually call. The `projects.$projectId.tsx` action keeps using the raw variants throughout (e.g. `deleteUploadedFile` needs the raw key, not a signed URL).

### Media storage

`app/services/storage/` is the storage abstraction all product-image/video read-write paths go through — no application code should assume media lives at `process.cwd()/uploads` directly (the local driver and the `/uploads/*` route are the only two places that do). `getMediaStorage()` (`services/storage/media-storage.server.ts`) picks a driver via `MEDIA_STORAGE_DRIVER` (`local` default or `oss`) and caches it as a singleton, same pattern as `getVideoQueue()`. Both drivers implement the same `MediaStorageDriver` interface (`services/storage/types.ts`): `saveBuffer`/`saveFromPath`/`readBuffer`/`readAsDataUrl`/`delete`/`resolveUrl`/`healthCheck`.

Object keys (`services/storage/keys.ts#buildObjectKey`) double as both the OSS object key and the local relative path under `uploads/` — one string format for both drivers: `product-images/<uuid>.<ext>`, `scene-videos/<projectId>/<uuid>.mp4`, `final-videos/<projectId>/<uuid>.mp4`. `isManagedRef()` distinguishes these (plus the legacy `/uploads/...` shape) from arbitrary external URLs (raw Wan/TTS provider URLs mid-flight, before they're copied into our own storage) — external URLs always go through a real network fetch with a timeout and a maximum size (`MEDIA_DOWNLOAD_TIMEOUT_MS`/`MEDIA_DOWNLOAD_MAX_BYTES`, both optional with built-in defaults), never a storage read.

`services/storage/local-storage.server.ts` stores files under `uploads/` (gitignored), guards path traversal by resolving every ref to an absolute path and checking it stays within the upload root (`path.resolve` + prefix check — not a naive string search, since canonical keys legitimately contain `/`), and serves reads back through `routes/uploads.$.tsx` (a splat route, since nested category keys contain slashes) regardless of the currently active driver, so a stray legacy `/uploads/...` reference still resolves while running in local mode. `services/storage/oss-storage.server.ts` uses the official `ali-oss` SDK; objects are private by default with signed URLs generated fresh per `resolveUrl()` call (never persisted), or public-base-URL mode if `OSS_PUBLIC_BASE_URL` is set. `createOssStorageDriver()` accepts an injectable client for testing (`app/services/storage/oss-storage.test.ts`), so OSS behavior is unit-tested against a fake client rather than a real bucket.

`scripts/video-worker.mjs` cannot import these TS modules (see the note on that file above), so `scripts/lib/media-storage.mjs` is a hand-kept-in-sync plain-JS mirror covering the subset it needs (save/read/delete, key building, the managed-vs-external-URL distinction, and the timeout/size-capped remote download). If you change key format, path-traversal logic, or OSS wiring in one, change both.

Switching an existing `local`-mode deployment to `MEDIA_STORAGE_DRIVER=oss` does not retroactively migrate already-uploaded files — legacy `/uploads/...` references only keep resolving as long as that deployment stays on `local` mode. No Drizzle migration was needed to add OSS support: the columns already held free-form strings, and only the string *format* of new writes changed.

### Types

`app/types/showrunner.ts` is the shared contract (`ProductBrief`, `ProductAnalysis`, `StoryPackage`, `DirectedScene`, `StoryboardScene`, `StoryBible`, `AgentTokenUsage`, `EditorPackage`, `ShowPlan`) that the Qwen agents and route UIs depend on — check this file first when changing the shape of generated data. `ShowPlan.analysis` and `ShowPlan.tokenUsage` are both optional so projects saved before those fields existed still parse fine.

### Route conventions

Routes are registered explicitly in `app/routes.ts` (not filesystem-based) using `@react-router/dev/routes`. `.server.ts` suffixes mark server-only modules, agents, and services that must not be imported from client-rendered code.
