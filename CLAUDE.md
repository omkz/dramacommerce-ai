# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

DramaCommerce AI is a React Router 8 full-stack app. A merchant uploads a product image plus a short brief (audience, mood, platform, duration); the app runs a Qwen-powered AI "showrunner" pipeline — Analyze (vision) → Story → Director → Prompt → Critic → Editor — that produces a story concept, hook, voice-over, a 5-scene storyboard with video prompts, an editing timeline, a social caption/CTA, can kick off real video renders for each scene via Alibaba Cloud's Wan model (text-to-video by default, or image-to-video using the real product photo for scenes the Director Agent marks as reference-eligible; each scene's AI-written voice-over line is synthesized via DashScope TTS and muxed onto the clip), and stitches the 5 successful clips into one final drama ad with ffmpeg.

## Commands

Package manager is **pnpm** (see `pnpm-lock.yaml` and the Dockerfile).

- `pnpm dev` — start the dev server (React Router dev, HMR) at `http://localhost:5173`
- `pnpm run typecheck` — runs `react-router typegen` then `tsc` (regenerate route types before type-checking after changing `app/routes.ts` or route loader/action signatures)
- `pnpm run build` — production build
- `pnpm run start` — serve the production build (`react-router-serve ./build/server/index.js`)

There is no test suite and no lint script configured in `package.json`.

`scripts/video-worker.mjs` (`pnpm run worker:video`) shells out to `ffmpeg` for the final-video stitch step — it must be installed and on `PATH` wherever the worker runs (already handled in the Docker image via `apk add ffmpeg`, but needs a manual install for local dev).

`scripts/showrunner-worker.mts` (`pnpm run worker:showrunner`) runs the Story/Director/Prompt/Editor agent pipeline as a background job. Unlike `video-worker.mjs` (plain `.mjs`, duplicates Wan/TTS API calls in raw JS), this worker imports the app's real TS modules directly (`services/showrunner.server.ts`, `services/project-store.server.ts`) via the `~/*` path alias — no duplicated agent logic. In development, `pnpm run worker:showrunner` runs the `.mts` source directly via `tsx`. In production, the Docker build instead runs `pnpm run build:worker` (esbuild, bundling the worker and its `~/*`-aliased imports — resolved from `tsconfig.json`'s `paths` — into one file, `build/worker/showrunner-worker.mjs`, with `node_modules` packages left external), and the container runs `pnpm run start:worker`, a plain `node` invocation with no `tsx`/TypeScript/`app/` source needed at runtime. `tsx` and `esbuild` are both devDependencies for this reason — only `pnpm run build`'s output ships to production, never the compiler/loader itself.

## Environment

Copy `.env.example` to `.env`. Three separate Alibaba Cloud DashScope surfaces are used, each with its own base URL var (even though TTS and Wan usually share the same workspace URL in practice):

- `DASHSCOPE_API_KEY`, `QWEN_BASE_URL` (includes `/compatible-mode/v1` — OpenAI-compatible chat completions), `QWEN_MODEL` — for the showrunner planning step. `QWEN_VISION_MODEL` (default `qwen3-vl-flash`, same base URL/key) is used only by the Analyze Agent's vision call.
- `DASHSCOPE_VIDEO_BASE_URL` (no `/compatible-mode/v1` suffix — uses `/api/v1/services/...` and `/api/v1/tasks/...`), `WAN_VIDEO_MODEL`, `WAN_VIDEO_RESOLUTION`, `WAN_VIDEO_RATIO`, `WAN_VIDEO_DURATION` — for Wan text-to-video generation. `WAN_VIDEO_I2V_MODEL` (default `wan2.1-i2v-turbo`) is used instead, on the **same endpoint**, for scenes where `useProductReference` is true — the request swaps `prompt`-only input for `{prompt, img_url}` and drops the `ratio` param (i2v derives aspect ratio from the reference image).
- `DASHSCOPE_TTS_BASE_URL` (same `/api/v1/services/...` style), `DASHSCOPE_TTS_MODEL` (default `qwen3-tts-flash`), `DASHSCOPE_TTS_VOICE` (default `Cherry`) — for scene voice-over synthesis. Uses the same `DASHSCOPE_API_KEY`.

If Qwen env vars are missing or the Qwen call fails, generation returns an error and no project is created. Do not reintroduce automatic mock fallback for production generation.

## Architecture

### Pipeline: brief → showrunner (async) → project → video

The full pipeline is visualized as a live "Agent Timeline" spanning two pages: `Analyze → Story → Director → Prompt → Critic → Editor` on `/projects/new/:jobId`, then `Render → Stitch` on `/projects/:id`. Both halves follow the same shape — enqueue a BullMQ job, a worker updates a Postgres status column stage by stage, the page polls and re-renders via the shared `app/components/agent-timeline.tsx` — because that's the only way to show live per-stage progress without inventing streaming infra this codebase doesn't otherwise use. `app/components/agent-timeline.tsx` also supports an `orientation` prop (`"vertical"` default, `"horizontal"`) — `/projects/:id` renders it horizontally in a full-width section above the two-column layout; `/projects/new/:jobId` keeps the original vertical layout.

1. `routes/projects.new.tsx` (`action`) collects the form/multipart submission, saves the uploaded image via `services/image-upload.server.ts`, creates a `showrunner_jobs` row via `services/project-store.server.ts#createShowrunnerJob`, enqueues a `showrunner.generate` job (`services/showrunner-queue.server.ts`), and redirects to `/projects/new/:jobId` — it does **not** call `generateShowPlan` itself.
2. `scripts/showrunner-worker.mts` (`pnpm run worker:showrunner`) picks up the job and calls `services/showrunner.server.ts#generateShowPlan(brief, onStageChange)`, which orchestrates six Qwen-powered stages in order, invoking `onStageChange` before each one so the worker can write the current stage (`ANALYZING` / `STORY` / `DIRECTING` / `PROMPTING` / `CRITIQUING` / `EDITING`) to `showrunner_jobs.status`. Each agent validates its JSON through `services/showrunner-validator.server.ts` before the next stage runs:
   - `agents/analyze-agent.server.ts` — a vision call (`services/qwen.server.ts#callQwenVisionJson`, `QWEN_VISION_MODEL`) on the uploaded photo, base64-inlined from local disk (`services/image-upload.server.ts#readUploadedImageAsDataUrl` — required even in production, DashScope fetches by URL or accepts inline data, and inlining avoids ever needing the upload to be publicly reachable). Returns category/colors/material/branding/quality/`canUseAsReference`/issues; feeds into Story and Director as grounding context.
   - `agents/story-agent.server.ts` — brief + analysis → `StoryPackage` (concept, hook, voice-over). Right after this stage, `services/story-bible.server.ts#buildStoryBible(brief, analysis, story)` condenses brief+analysis+story into a compact `StoryBible` (`productFacts`/`visualStyle`/`storyCore`/`constraints`) that every downstream agent below reads instead of the raw objects — those don't need fields like `imageUrl`/`showProductOverlay`, and re-serializing all three full objects into five separate prompts was pure repeated context. Raw `brief`/`analysis` are still passed into the agent functions where needed for local skill computation (`services/skills/*`), which costs no tokens since it's not sent to Qwen.
   - `agents/director-agent.server.ts` — brief + analysis + Story Bible → `DirectedScene[]` (5 fixed scenes, each with `camera`/`emotion`/`useProductReference`). `useProductReference` may only be true when `analysis.canUseAsReference` (`storyBible.visualStyle.canUseAsReference` in the prompt) is true **and** the scene's composition could plausibly continue from the actual photo (normally just the final reveal/hero scene) — Wan i2v uses the photo as a literal first frame, so forcing it onto an unrelated b-roll shot produces broken video.
   - `agents/prompt-agent.server.ts` — scenes + Story Bible → `StoryboardScene[]` (adds a Wan-ready `videoPrompt` per scene). Uses Qwen function-calling (`services/qwen.server.ts#callQwenJson`'s `tools`/`toolHandlers` params) to call a local `get_wan_video_constraints` tool for the real configured resolution/ratio/duration before writing prompts, instead of guessing. Note: Qwen's OpenAI-compatible endpoint hangs indefinitely if `response_format: json_object` is combined with `tools` in the same request (confirmed live) — `callQwenJson` only sets `response_format` on tool-free calls. Accepts an optional `revisionNotes` param for the Critic's revision pass below.
   - `agents/critic-agent.server.ts` — storyboard + Story Bible (+ raw `analysis` for its own skill checks) reviews the storyboard for pacing/coherence and sanity-checks `useProductReference` choices against each scene's actual `videoPrompt`. Returns `{approved, notes?}`; if not approved, `generateShowPlan` re-runs the Prompt Agent once with the notes — bounded to exactly one revision pass, no second critique call.
   - `agents/editor-agent.server.ts` — storyboard + Story Bible → `EditorPackage` (timeline, caption, CTA). Reads `storyCore.hook` from the Story Bible and is instructed to open/riff on it in the caption, so the caption doesn't drift to a different angle than the Hook the merchant sees and can edit.
3. On success the worker calls `saveProject` and marks the job `SUCCEEDED` with `projectId` set; `/projects/new/:jobId`'s loader redirects to `/projects/:id` once it sees that. On failure (BullMQ retries exhausted) the worker marks the job `FAILED` with a friendly error via `services/qwen.server.ts#getQwenErrorMessage` and deletes the uploaded image (mirrors the old synchronous failure cleanup) — no project is created either way.
4. `routes/projects.tsx` lists saved projects; `routes/projects.$projectId.tsx` shows one project's full plan and drives Wan video generation for any/all scenes (`intent=create-video-task` / `intent=refresh-video-task` / `intent=create-all-video-tasks` form actions call `services/wan-video.server.ts`, job state stored via `services/project-store.server.ts#saveVideoJob`, including each scene's `useProductReference` flag). `scripts/video-worker.mjs` reads that flag: if true, it reads the local upload, base64-encodes it, and calls Wan with `WAN_VIDEO_I2V_MODEL` + `img_url` instead of `WAN_VIDEO_MODEL` + `prompt`-only. When it polls a scene to `SUCCEEDED`, it synthesizes that scene's `voiceOver` line via DashScope TTS and muxes the audio onto the clip with `ffmpeg` before saving — the raw Wan clip has no audio track at all. If TTS or muxing fails, the worker falls back to the silent Wan clip rather than failing the job (the Wan generation cost is already spent). The product-image overlay (small watermark-style thumbnail in the corner) is only applied when `useProductReference` is false — a reference-image scene already has the real photo baked into frame one, so stamping it on top again would be redundant — and only when the merchant hasn't opted out via the `showProductOverlay` checkbox on `/projects/new` (`ProductBrief.showProductOverlay`, threaded through `VideoCreateJobData.showOverlay`; `undefined`/missing is treated as `true` for backward compatibility with projects saved before this field existed). Once all 5 scenes reach `SUCCEEDED`, `intent=create-stitch-task` enqueues a `video.stitch` BullMQ job; the worker downloads the 5 (now-narrated) clips, concatenates them with `ffmpeg` (stream-copy first, re-encode fallback if codecs mismatch), and writes the result to `uploads/`. Stitch status/output is tracked in the `final_videos` table (one row per project) via `services/project-store.server.ts#saveFinalVideo`.

Every `callQwenJson`/`callQwenVisionJson` call (`services/qwen.server.ts`) accepts an optional `onUsage` callback reporting `{promptTokens, completionTokens, totalTokens}`, summed across BullMQ tool-calling rounds. `generateShowPlan` tags each call with its stage/model and collects them into `ShowPlan.tokenUsage` (optional field, no migration needed — `show_plan` is jsonb), rendered as a table on `/projects/:id`.

`wan-video.server.ts#createWanTextToVideoTask` (the TS version) is currently unused dead code — the actual Wan task creation logic (including the i2v branch above) lives only in `scripts/video-worker.mjs`, which duplicates it in plain JS rather than importing app services (a deliberate, pre-existing pattern for that worker, not something this pipeline change altered).

Editor Agent's `timeline`/`caption`/`cta` output is still display-only — the actual `ffmpeg` stitch step is a plain concat and does not burn in subtitles or append an end-of-video packshot from `EditorPackage`. That integration (making the render actually consume Editor Agent's output) is a deliberately separate, not-yet-started follow-up.

### Storage

`services/project-store.server.ts` persists projects, showrunner jobs, and video jobs to Postgres through Drizzle (`app/db/schema.ts`, `services/db.server.ts`). Migrations live in `drizzle/` and should be applied with `pnpm run db:migrate` before starting web or worker processes. `projects.show_plan` is stored as JSONB, `showrunner_jobs` stores the pending brief (`brief_json`), current pipeline status, error, and the resulting `project_id` once saved, `video_jobs` stores per-scene provider, queue job ID, Wan task ID, status, attempts, polling timestamps, output URL, and error metadata, and `final_videos` stores the stitched-output status/URL per project. Uploaded product images and stitched final videos are written to `uploads/` on local disk and served back through `routes/uploads.$filename.tsx`, which guards against path traversal by rejecting filenames containing `/` or `..`. `uploads/` is gitignored and should move to OSS for production scale.

### Types

`app/types/showrunner.ts` is the shared contract (`ProductBrief`, `ProductAnalysis`, `StoryPackage`, `DirectedScene`, `StoryboardScene`, `StoryBible`, `AgentTokenUsage`, `EditorPackage`, `ShowPlan`) that the Qwen agents and route UIs depend on — check this file first when changing the shape of generated data. `ShowPlan.analysis` and `ShowPlan.tokenUsage` are both optional so projects saved before those fields existed still parse fine.

### Route conventions

Routes are registered explicitly in `app/routes.ts` (not filesystem-based) using `@react-router/dev/routes`. `.server.ts` suffixes mark server-only modules, agents, and services that must not be imported from client-rendered code.
